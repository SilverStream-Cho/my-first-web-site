20. 마무리

더 많은 것을 알고 싶고, 가야 할 길이 멀기 때문에 마음이 급할 것이다 그런데 지금은 공부를 더 하기도 좋은 때이기도 하지만, 공부를 멈추고 공부한 것을 사용하기도 좋은 때이다. 공부만 하고 공부한 것을 사용하지 않으면 나중엔 머리 속이 너무 복잡해져서 막상 코딩을 하려니 무엇을 해야할지 모르는 상태가 될 수 있다.
좋은 코드가 무엇인지는 아는데, 좋은 코드를 짤 줄 모른다면 내가 짠 코드가 실망스럽게 된다 이래서는 좋은 생산자가 될 수 없다 현명한 판단을 할 수 있도록 생각의 재료를 쌓아보자 


    <input type="checkbox">
그러면, 우리가 체크박스를 이용해서 어떠한 앱을 만들었다고 해보자 그러면 체크박스가 하나면, 우리가 만든 앱이 잘 동작하는지 확인하기 위해서 몇 번의 테스트가 필요할까 2번이다
  • 체크를 안한 상태에서 실행시켜서 테스트를 해보고,
  • 체크를 해본 상태에서 실행시켜서 테스트를 해봐서 문제가 없는지를 체크해봐야 한다
그러면 체크박스가 두 개면 어떻게 할까 

        <input type="checkbox">
        <input type="checkbox">
4번의 테스트가 필요하다
  • 체크를 둘다 안 한 상태에서 실행
  • 체크를 둘다 한 상태에서 실행
  • 체크를 왼쪽만 한 상태에서 실행
  • 체크를 오른쪽만 한 상태에서 실행
그러면 체크박스가 세 개면 어떻게 할까 

        <input type="checkbox">
        <input type="checkbox">
        <input type="checkbox">
8번의 테스트가 필요하다
  • 체크를 둘다 안 한 상태에서 실행
  • 체크를 둘다 한 상태에서 실행
  • 체크를 첫번째만 한 상태에서 실행
  • 체크를 두번째만 한 상태에서 실행
  • 체크를 세번째만 한 상태에서 실행

  • 체크를 1, 2번째만 한 상태에서 실행
  • 체크를 1, 3번째만 한 상태에서 실행
  • 체크를 2, 3번째만 한 상태에서 실행
만약 체크박스가 50개라면?
  • 직관적으로 생각해보자(수학적으로 생각하지 말고..)
  • 사실 수학적으로 사고하면 2의 50제곱이 우리가 원하는 값이다
  • 이건 실제로 얼마일까? (1천조이다)

우리 주변에는 체크박스 50개보다 복잡한 앱들이 얼마나 많은데, 세상은 우리 생각과는 다르게 아주 쉽게 복잡해진다 눈에는 체크박스 50개가 보이기 때문에 쉽다고 생각하지만, 실제로는 눈에 보이지 않는 천조 가지의 경우의 수가 숨어있을 때, 테스트해보지 못한 경우에서 예상못한 버그들이 생겨나게 된다 또 작은 기능을 추가하는 것도 나중에는 어렵게 되면서 깊은 슬럼프에 빠진다

이 복잡함은 눈에 보이지 않기 때문에 자연스럽게 좌절감의 원인을 숨어있는 복잡함 때문이 아니라 자기의 탓으로 돌리게 된다 그러나 원래 그렇다면 공부도 마찬가지이다 개념을 알게 된다는 것은 말하자면 체크박스가 늘어난다는 것이다 자연스럽게 머릿속이 기하급수적으로 복잡해진다 그러면 이 복잡함은 나쁜 것일까? 오히려 인류는 눈에 보이는 단순함을 조작해서, 눈에 보이지 않는 엄청나게 복잡한 것을 만들어내는 위대한 존재이다

가장 쉬운 예시가 언어이다. 우리는 주어, 동사, 목적어라는 쉬운 덩어리들로 해서 셀 수 없는 의미들을 만들어낼 수 있다 문법이 조금만 복잡해져도 이득은 천문학적으로 커진다 어순만으로도 우리는 엄청나게 많은 의미를 쏟아낼 수 있다 이렇게 간단한 문법이 이렇게 대단한 도구라는 것을 알았다면 이것들만 가지고 의미를 만들어보는 연습을 충분히 했을 것이다 그러나 이렇게 중요한 도구의 사용에 충분히 익숙해지기도 전에 더 복잡한 문법들을 배우다보니 머릿속이 너무 복잡해져서, 한 마디도 할 수 없게 된다

지금까지 배운 것도 마찬가지이다. 영어의 주어 동사 목적어처럼 혁명적인 것이다 이것만으로도 우리는 웹이 웹이기 위해서 필요한 모든 것을 알게 되었다 덜 아름답고 덜 편리하고 덜 생산적일 뿐이다

그러면 어떻게 해야할까, 진도를 멈추고, 나의 삶에서 정말 중요한 정보를 정리정돈해보자 그것이 나의 인생에서 중요한 것일수록 여러 가지 불만들이 쏟아져 나올 것이다 더 예쁘게 만들 수도 있고, 더 편리하게 만들고 싶을 수도 있을 것이다 또, 더 생산적으로 웹페이지를 만들어내고 싶을 수도 있다 중요한 것은, 이러한 불만족을 자기 눈으로 직접 확인하는 것이다 그래야 부분학습할 수 있다 또, 그 불만족이 누적돼서 절망감이 충분히 성숙해야 한다 그래야 공부가 우리 문제를 해결해주는 구원자가 되는 것이다 바로 이때가 진도를 나갈 절호의 타이밍이다 더 빨라져도, 더 늦어져도 안 된다

무언가를 만들어보면서 한계에 스스로 직면해보자 그리고 그 한계에 대해서 충분히 절망하기를 바란다 그 절망감이 충분히 성숙했을 때, 절망감을 극복하기 위한 방법이 마련되어 있으니 학습을 하면 된다


본질에서 벗어나 혁신을 향해가기

학습의 주인공이 되어서 공부의 방향을 스스로 선택해야 한다 공부의 효용은 최소한으로 배워서 최대한 써먹을 때 드러난다고 생각한다 최대한으로 배워서 최소한으로 써먹으면 공부와 멀어지는 지름길이다

이제 우리의 진로에 대해서 곰곰히 따져보자

  1. 만일 웹사이트를 아름답게 하고 싶다면
    • css를 공부해야 한다
    • css는 html이 태어난지 약 4년 후에 만들어지기 시작했다
    • 웹 퍼블리셔, 웹디자이너와 같은 직업이 활성화되기 시작했다
  2. 만일 사용자와 상호작용하는 웹페이지를 만들고 싶다면
    • js를 공부해야 한다
    • js는 html이 태어난지 약 5년 후에 만들어지기 시작했다
    • web front end engineer, web interactive designer가 발전하기 시작했다
만약 우리가 1억 개의 웹페이지로 만들어진 웹사이트를 운영하고 있다고 해보자 그런데 어떤 이유로 웹사이트에 있는 웹페이지에서 'h1'태그를 'h2'태그로 변경해야한다면, 1억개의 웹페이지를 변경해야 한다 하나의 페이지를 변경하는데 0.1초가 걸린다고 해도 1억개면 115일이 걸린다 생산성의 한계에 봉착하는 것이다 이 한계를 극복하기 위해서는 백엔드 기술을 배워야 한다 이 기술을 이용하면, 하나의 파일을 변경하면 1억개의 웹페이지가 동시에 바뀌는, 폭발적인 효과를 경험할 수 있다
  • PHP
  • JSP
  • Node.js
  • 파이썬의 장고
  • 루비의 루비온레일즈
  • ASP.NET
이런 기술들은 바로 이런 목적을 위해서 고안된 기술들이다 그리고 이 기술들은 서로 경쟁관계에 있기 때문에 이 중에 하나를 공부하면 된다 이 기술을 위해서 종사하는 엔지니어를 Backend engineer라고 부른다

만약 웹사이트로 돈을 벌고 싶다면, 웹페이지에 광고를 달아서 이 광고에 노출에 따라서 돈을 벌 수도 있다 그러면 web2 광고 수업을 들으면 된다

web2 중에 자신이 필요한 것을 공부하면 된다 앞으로 웹1을 뿌리로 하는 다양한 것을 수강할 수 있도록 할 것이다
웹 프로젝트에서 가장 중요하게 생각하는 것은, 누구나 웹 1을 기반으로 새로운 수업을 만들 수 있기 때문이다 web1은 저작권이 없는 공공재로 만들어졌다

알고 있는 지식을 공고히 하는 방법

  1. 알고 있는 지식을 활용해서 프로젝트를 하는 것
  2. 알고 있는 지식을 컨텐츠로 만들어서 누군가에게 알려주는 것
둘다 자신이 무엇을 알고 있는지를 자신과 타인에게 알려줄 수 있는 너무나 좋고 중요한 방법이다 더 늦기 전에 프로젝트를 시작하고, 나만의 컨텐츠를 만들어내자
