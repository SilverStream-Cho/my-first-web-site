34. 파일로 쪼개서 정리정돈하기

우리 지난 시간에 정리정돈의 도구들, 함수 배웠고, 객체 배웠다. 이번 시간에 이것보다 더 큰 정리정돈의 도구, 어떻게 보면 가장 큰 정리정돈의 도구가 아닐까 싶은 기능에 대해서 살펴볼 것이다.

서로 연관된 코드들을 파일로 묶어서 그룹핑하는 것이다. 이것을 사용하면, 우리가 웹페이지가 아무리 많아도, 그 많은 웹페이지들을 우리가 감당할 수 있게 된다. 


        <input type="button" value="night" onclick="
        nightDayHandler(self);
       ">
이 태그를 모든 페이지에 복붙해서 적용해놓으면, 모든 페이지에 버튼이 생긴다. 그러나 클릭해도 작동이 안 된다. 왜? js는 안 따라갔으니까. 그러면 script 태그를 그대로 카피해서 모든 웹페이지에 배포하자. 그러면 이제 잘 동작한다.

그러면, 이제 극단적인 생각을 좀 해보자. 우리가 갖고 있는 웹페이지가 1억개라면? 배포도 어렵긴 하겠지만, 만약 우리가 지금까지 코딩했던 것에서, powderblue를 yellow로 바꾸고 싶다. 그러면 어떻게 해야하나? 이 수정한 것을 모든 웹페이지에서 바꿔야 한다. 어렵다. 바로 이러한 절망적인 상황에서 우리를 구출해주는 도구, 바로 파일로 쪼개는 것이다. 일단은 파일을 만들자. color.js라는 파일을 만들 것이다. (컬러와 관련된 기능들이니) 그리고 3.html에서 모든 웹페이지에 공통적으로 들어가는 공통된 코드를 script 태그를 제외하고 카피해서 붙여넣는다.

그다음에 공통된 코드를 지우고, script 태그에 src 속성(source의 약자)으로, 'color.js'를 지정하고 리로드하면 그대로 동작하는 것을 알 수 있다. 


        <script src="colors.js"></script>
하지만 외부적인 구현방법은 완전히 달라진 상태이다. 어떤 상태? 파일로 쪼개진 상태이다. (이전에 css 파일을 분리했던 것과 유사하다.)


35. 라이브러리와 프레임워크

이제 마지막 수업을 시작해보도록 하겠다. 오늘날 우리가 소프트웨어를 만든다고 하면, 혼자 만드는 경우는 없다. 그렇게 할 수가 없다. 왜냐하면 우리는 누군가가 만든 소프트웨어 위에서 동작하는 소프트웨어를 만들고 있는 것이기 때문이다. (웹브라우저처럼) 그리고 그런 얘기가 아니라고 하더라도, 우리가 소프트웨어를 만들 때, 자기 혼자 만드는 것보다는 다른 사람이 이미 잘 만들어놓은 것을 부품으로 해서, 내가 만들고자 하는 것을 빠르게 조립해서 만들어가는 것, 그것이 오늘날 소프트웨어를 만드는 아주 기본중의 기본이라고 할 수 있겠다. 그래서 지금까지는 생산자가 되는 방법을 살펴보았고, 이제부터는 다른 사람이 만든 소프트웨어를 부품으로 해서 그 소프트웨어를 소비해서 내가 만들고 있는 소프트웨어의 생산자가 되는 방법을 살펴보려고 한다. 다시 말해서 생산자로서의 소비자, 그런 것들을 살펴본다. 소프트웨어의 사회성이라고 할 수 있을 것이다.

이때 우리가 알아두면 좋을 용어는 두 가지이다. library와 framework이다. 두 가지는 비슷하다. 다른 사람의 도움을 받아서 소프트웨어를 만든다. 라는 점에서 비슷하다. 라이브러리는, 도서관이라는 뜻이다. 이는 무언가가 정리정돈되어있는 곳을 의미한다. 즉, 내가 만들고자 하는 프로그램에 필요한 부품들이 되는 소프트웨어를 잘 정리정돈해놓은, 재사용하기 쉽도록 되어있는, 그러한 소프트웨어를 라이브러리라고 한다. 개인적인 느낌은 내가 만들고 있는 프로그램에 사용하는 부품을 가져오는 느낌이다.

그리고 프레임워크라는 개념이 있다. (두 가지를 똑같이 봐도 상관이 없긴 한데. 약간 뉘앙스가 다르다.) 프레임웤은 우리가 만들고자 하는 것이 있을 떄, 그 만들고자 하는 것이 무엇이냐에 따라서, 예를 들면 게임이냐. 웹이냐. 채팅이냐. 여러 가지가 있는데, 그것을 만들려고 할 때, 언제나 필요한 공통적인 것이 있고, 또 우리가 만들고자 하는 기획의도에 따라서 달라지는 부분들이 있을 것이다. 그중에서 공통적인 부분은, 프레임웤이라는 부분을 만들어놓고, 우리는 만들고자 하는 것의 기능에 따라서, 또는 개성에 따라 달라지는 부분만 살짝씩 수정하는 것을 통해서 우리가 만들고자 하는 것을 처음부터 끝까지 만들지 않도록 해주는, 거의 반제품과 같은 것이 프레임웤이다.

라이브러리는 소프트웨어를 만드는 내가 라이브러리를 땡겨와서 쓰는 것의 느낌이라면, 프레임웤은 프레임웤 안에 들어가서 우리가 작업하는 느낌이다. 어쨌든 무엇이 되었건, 다른 사람과 협력하는 모델이라고 이해하면 된다.

그중에 라이브러리를 살펴볼 것인데, js 라이브러리중에, 가장 유명한 것은 바로 jQuery라는 라이브러리이다. 이는 상당히 오래되었고, 아주 안정적이다. 그리고 이것을 사용하면, 이것을 사용하지 않고 직접 코딩했던 것보다 생산성이 훨씬 더 높아진다. 우리가 배웠던 지식이 있어야 제이쿼리를 잘 다룰 수 있다. 또 우리가 지금까지 다루었던 지식만 가지고 우리가 느꼈던 절망감이 있어야 제이쿼리를 만났을 때 즐거워진다. 즐거워져 보자.

제이쿼리를 검색해보자. jquery 홈페이지 그러면 download 탭에서 제이쿼리를 다운로드받을 수 있는 방법들이 들어있다. 링크를 클릭하면 다운로드가 시작이 된다. 다운로드 된 파일을 프로젝트 폴더 디렉토리에 이동해도 된다. 편리한 방법은, CDN이라고 찾아보면 된다. (content delivery network의 약자) 그래서 직접 이 라이브러리를 다운로드 받아서 자기 프로젝트에 포함시키고, 업로드하고, 그걸 또 서비스하려면 돈 들고, 복잡하다. 그래서 많은 라이브러리들이 CDN이라는 것을 통해서 자기들의 서버에다 파일을 보관해놓고, 우리는 script src를 통해서 가져갈 수 있도록 하는 방식을 취하고 있다. 사용력까지 자기들이 대준다. (좋은 세상이다.) 그중에서 (잘 찾아봐야 한다. 내용은 달라질 수 있다.) 아래에 보면 구글 CDN이 있다. 눌러서 들어가보자.

여기 보면 google에서 제공하는 jQuery의 최신버전 코드가 있는데, 이것을 카피하자. 그리고 3.html로 돌아온 다음에, colors.js앞에 붙여넣자.
이 코드는, script 태그이다. src를 통해서 구글에서 제공하는 제이쿼리라고 하는 라이브러리의 주소가 적혀있다. 그러면 이제 끝이다. 더이상 뭘 다운로드 받을 필요가 없다. 이 상태에서 제이쿼리를 활용해서 뭔가를 해보겠다. (물론 나는 사용설명서를 좀 봐야 한다.)

제이쿼리를 활용해서 colors.js 파일을 수정해보겠다. 제이쿼리 이용했을 때 가장 놀랐던 것은, 어떤 처리해야 할 태그들이 여러 가지가 있을 때, 반복문을 통해서 처리해야 한다. 그런데, 제이쿼리를 이용하면 반복문을 쓰지 않아도 된다. 반복문을 우리 대신에 처리해주니까. 우리는 그냥 한 줄짜리 코드로 이 일을 할 수 있다. 


            var Links = {
                setColor: function (color){
                var alist = document.querySelectorAll('a');
                               var i = 0;
                               while(i < alist.length){
                                   alist[i].style.color = 'powderblue';
                                   i = i + 1;
                                 }
                                }
            }
어느 부분은 주석처리해서 동작하지 않도록 해보겠다. ctrl+/를 하면, 많은 에디터들이 주석을 만들어준다. 

            var Links = {
                setColor: function (color){
                
                                 $('a').css('color', color);
                                }
            }
제이쿼리는 $로 시작한다. $()라는 이름의 함수이다. 이것은 이 웹페이지에 있는 모든 a태그를 제이쿼리로 제어하겠다는 뜻이다. 그리고 제이쿼리에서, (jquery css) css를 바꾸려고 하면, '.css()' css라는 함수를 쓰면 된다고 한다. 제이쿼리를 이용해서 우리가 css를 어떤 element(태그)의 css를 바꾸고 싶을 때에는 .css를 쓴다. 두번째 인자로는 매개변수를 통해 들어온 'color'값을 준다. reload하면, 이 링크들이 흰색이 된 것은, 이 코드가 잘 동작했다는 뜻이다. 만약 이 코드를 주석처리하면? 되지 않는다.

위에 우리가 전에 코딩해두었던 코드와 지금 이것, 어느 것이 더 코딩하기 쉬울까? 뭐가 더 직관적인가? 제이쿼리가 훨씬 직관적이다. 그러나 한 가지 기억해야할 것은, 제이쿼리는 새로운 언어가 아니고, 자바스크립트를 이용해서, 우리 대신에 css라는 함수를 제이쿼리가 만들어둔 것이다. 우리는 저 css라는 함수를 사용하면, 저 함수가 내부적으로 우리 대신에 처리해주고 있는 것임(주석처리한 코드들)을 음미했으면 좋겠다.

나머지 코드들도 바꿔보자. 


        var Body = {
            setColor: function (color){
                
                $('body').css('color', color);
            },
            setBackgraounColor: function (color) {
                
                $('body').css('backgroundColor', color);
            }
        }

환상적이다. 자바스크립트는 오늘날 가장 빠른 속도로 성장하는 언어 중 하나이고, 또 중요한 시스템이기 때문에 수많은 라이브러리들이 쏟아지고 있다. 그래서 문법을 잘 알고, 하는 것도 중요하지만 세상에는 어떤 라이브러리가 새로 생겼는가. 어떤 라이브러리들이 있는가를 많이 알수록 나는 많은 일을 할 수 있는 가능성을 갖게 된다. 우리가 작업한 내용을 완성하자. 제이쿼리를 웹페이지에 삽입한 코드를 모든 페이지에 붙여넣기 한다. 그러면 페이지마다 잘 동작하는 것을 볼 수 있다.

이렇게 해서 파일로 로직을 쪼갰을 떄, 자기가 프로그램을 짤 때도 도움이 되지만, 다른 사람이 정리정돈한 것을 나의 프로젝트로 가져올 때도 중요한 역할을 하는 것이 파일로 코드를 정리정돈하는 것이라는 것을 살펴보았다. js 커뮤니티에서 가장 중요한 라이브러리 중에 하나인 제이쿼리를 사용하는 중요한 방법도 살펴보았다.


36. UI VS API

우리는, JS의 핵심적인 문법을 살펴보았다. JS를 통해서 웹 어플리케이션이 개선되는 과정도 체험해보았다. 지금까지 여러분에게 숨겨두었던 정말 중요한 개념을 소개해드리고 물러날 것이다. UI와 API라는 것이다.

UI는 user interface의 약자이고, API는 application programming interface의 약자이다. 이 두 개의 서로 다른 듯, 비슷하고, 비슷한 듯 다른 말의 의미를 좀 따져가면서, 우리가 하는 일이 무엇인가를 좀 음미해보는 시간을 가져보도록 하겠다. 여기 버튼이 있다. 


        <input type="button" value="Click me" onclick="alert('hello world')">
이걸 누르면 경고창이 뜬다. 이 버튼은 누가 사용하고 있나? 우리의 앱을 이용하려고 하는 사용자가, 이런 버튼과 같은 조작장치를 이용해서 우리의 웹 애플리케이션을 사용하고 있는 것이다. 사용자가 시스템을 제어하기 위해서 사용하는 조작장치를 유저 인터페이스라고 부른다.

그다음에는 이번에는 우리가 만든 웹 어플리케이션의 코드를 보자. 여기있는 이 경고창은 우리가 만든 것인가? 우리가 만들지 않은 것인가? 우리가 만들기도, 만들지 않기도 했다. 생각해보면 이 경고창이 실행되는 타이밍이나, 경고창에 표시되는 텍스트는, 우리의 의도가 반영된 것이기 때문에 우리가 만든 것이라고 할 수 있다. 하지만 이 코드, 이 한줄 짜리 코드에는 이 경고창의 기능이나 모양이나, 이런 것들이 일체 설명되어 있지 않다. 그러면 이 경고창은, 어떻게 만들어진 것일까? 그것은 웹브라우저를 만들어낸 사람들이 우리 대신에 경고창 기능을 미리 만들어두었다가, 우리가 alert이라는 함수를 실행하면, 경고창을 띄워주겠다고 js에 사용설명서를 통해 약속한 것이다. 그러면 우리는 그 약속을 믿고, alert이라는 함수를 호출해서 경고창을 켤 수 있게 되는 것이다. 함수 alert은, 경고창을 실행하는, 조작장치인 것이다. 그런데 이 조작장치를 일반인이 사용하지 않는다. 일반인은 코딩을 하지 못한다고 치면, 바로 우리가 만들고 있는 버튼을 클릭하면 경고창이 뜨는 애플리케이션이, 웹브라우저가 갖고 있는 경고창의 기능을 alert이라고 하는 js의 문법에 따라, 그 문법의 코드를 이용해서 사용하고 있는 것이다. 이렇게 application을 만들기 위해서 프로그래밍을 할 때 사용하는 조작장치들을 application programming interface라고 한다. alert이라는 것이 바로 API인 것이다. 이것은 JS에 국한된 이야기가 아니고, 모든 프로그래밍 언어에 공통적으로 적용되는 이야기이다. 모든 애플리케이션들은, API를 프로그래밍적으로, 다시 말해서 순서대로 실행하는 것을 통해서 만들어진다. API가 없다면, 순서가 무슨 소용이겠는가. 또 순서가 없다면, API가 무슨 소용이 있겠나. API와 순서는 단어와 문법처럼, 부품과 그 부품의 결합방법처럼 뗄레야 뗄 수가 없는 관계이다.

프로그래머가 되기 전까지는 평생 UI만 사용했을 것이다. 이제 우리는 API도 사용하는 프로그래머가 된 것이다. 이제 JS라는, 말하자면 접착제로, 이 API들을 결합해서, 다시 말해서 응용해서, 세상에 누구도 만들어본 적이 없는 응용프로그램을 만들 수 있게 된다.

다음 시간에는 웹브라우저가 어떤 API를 감추고 있는지를 소개하면서 대단원의 막을 내릴 것이다.


37. 수업을 마치며

이제 필요한 것은 공부보다는 실습이다. 실습보다는 자신의 프로젝트를 실행하는 것이다. 공부에는 때가 없지만, 프로젝트를 시작하는 것에는 때가 있다. 지금 이 순간이, 내 프로젝트를 시작할 때이다. 시작이 늦어질수록, 공부를 많이 할수록 자기가 짠 코드를 긍정하기 어려워진다. 머릿속에 복잡해진다. 그래서 프로젝트를 시작하려는 시기에서 하고자 하는 얘기가 있다.

자신의 프로젝트를 시작할 때, 모든 개념을 총동원하려고 하지 말아라. 필수 불가결한, 최소한의 도구만을 가지고 문제를 해결하려고 시도해봤으면 좋겠다. 그러면 최소한의 도구는 무엇인가? 그것은 프로그래밍이라는 말 속에 이미 함의되어있다. 순서에 따라 실행되어야 하는 명령들이 실행되도록 하는 것이다. 이것이 처음 시작하는 우리에게 필요한 유일한 도구이다. 객체는 말할 것도 없고, 함수도, 조건문도, 반복문도 필요하지 않다. 순서대로 실행되는 것만으로도 프로그래밍은 충분히 혁명적이다. 머리가 복잡하면 아무것도 할 수 없다. 최소한의 지식으로 현실의 문제를 해결하기 위해서 노력해보자. 그러다보면, 그것만으로는 도저히 해결할 수 없는 순간이 오는데, 그때 주의깊게 반복문, 조건문, 함수를 신중하게 도입하면서 각각의 개념들 간의 관계에 익숙해지다보면 마침내 유창하게 이런 개념들을 활용할 수 있게 될 것이다. 그렇게 즐거운 시간을 보내다보면 또 다시 한계가 찾아올 것이다. 바로 그때가 실습을 멈추고 다시 공부를 시작할 때라고 생각한다. 앞으로 부딪히게 될만한 몇 가지 한계들과, 그 한계들을 극복할만한 검색어들을 추천하고 마무리하겠다.

웹페이지에 어떤 태그를 삭제하고 싶거나. 어떤 태그의 자식 태그를 추가하고 싶다면 document 객체를 좀 더 자세히 살펴보자. 내가 필요한 method가 들어있을 것이다. 만약에 document로도 찾을 수 없다면, DOM(document object medel)으로 수색범위를 넓혀보자. (document 객체가 이것의 일부이다.) 또 웹페이지가 아니라 웹브라우저 자체를 제어해야 한다면, window 객체를 조사해보자. 이를테면 현재 열려있는 웹페이지의 주소를 알아내야 할 수도 있고, 새 창을 열어야 할 수도 있고, 또 웹브라우저의 화면크기를 js를 통해서 알아야 한다면, window 객체에 속해있는 property나 method가 도움이 될 것이다. 또 웹페이지를 리로드하지 않고 정보를 변경하고 싶다면, ajax가 필요할 것이다. (현대적인 외벽을 만드는 데 필수적인 테크닉이다.) 또 웹페이지가 리로드되어도 현재 상태를 유지하고 싶다면, cookie를 배워라. 쿠키가 함께라면, 사용자를 위한 개인화된 서비스를 제공할 수 있다. 인터넷이 끊겨도 동작하는 웹페이지를 만들고 싶다면, offline web application을 찾아봐라. 화상통신 웹 앱을 만들고 싶다면 webRTC라는 것이 있다. 사용자의 음성을 인식하고, 음성으로 정보를 전달하고 싶다면 speech로 시작되는 API를 살펴보자. 그리고 3차원 그래픽으로 게임과 같은 것을 만들고 싶다면, webGL을 살펴보자. 가상현실에 관심이 많으면 webVR도 있다.
